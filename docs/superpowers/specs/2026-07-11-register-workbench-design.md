# 등록 워크벤치 — 표준 중심·자율·설명 내장 (등록/설정 UX 재설계)

날짜: 2026-07-11
상태: 설계 확정 (형 승인 — "표준화해놓고 유저 자율에 맡기되 과정을 편하게" + "x-marina·compose 몰라도 알아먹게"), 구현 전
선행: worktree 콘솔 1단계(카드·탭·질감·빌드로그) — 같은 브랜치(claude/worktree-console-ui)
근거 조사: 등록 플로우 UX 지도(탐색 에이전트, 2026-07-11) — 마찰 4지점 코드 확인

## 배경 — 팀원이 막히는 4지점 (코드 근거)

1. **진입 발견성**: 등록 버튼이 프로젝트 스위처 드롭다운 안에만 존재(app-1-core:78-82), 빈 상태 CTA 없음.
2. **경로/탐색기**: 브라우저 안 폴더 탐색기가 어색 + 위저드에서 경로 입력칸이 [다음] 버튼보다 아래 렌더(index.html DOM 순서).
3. **위저드 구조**: 4스텝·13클릭, 뒤로가기 시 선택 유실(카드 상태 미보존), Esc 즉시 유실, 고급(YAML) 편도행.
4. **용어·개념**: compose·x-marina·엮기·심링크·attach 를 모르면 체크 근거를 알 수 없음. 도움말이 hover 툴팁뿐.
5. **등록 후 단절**: 완료가 alert 지시문("카드에서 ▶로") — 다음 행동이 UI 로 이어지지 않음.

## 설계 원칙 (형 확정)

- **표준화 + 유저 자율**: marina 의 표준 = compose 파일 하나(services + x-marina). 등록 = 유저가 그 표준 문서를 직접 완성하는 것. **UI 는 아무것도 대신 결정하지 않는다** — 자동은 "재료 제안"까지만. (이유: 모든 자동 케이스를 marina 가 책임질 수 없음)
- **매끄러움 = 과정의 품질**: 좋은 에디터, 근거 있는 제안, 인라인 검증, 유실 없는 구조 — 마법이 아니라 도구.
- **설명 내장**: x-marina 옵션과 compose 기본 개념을 **몰라도 화면 안에서 배울 수 있게** — 문서 링크로 도망가지 않는다.
- **Orca 스킨 통일**: 등록·links·구성 모달 전부 카드와 같은 문법(무테두리·`--st-*` 팔레트·모노 코드·hover 액션).

## 비목표

- 자동 등록/요약 카드 원버튼(A안) — 폐기(형: 자동 지양).
- LLM 기반 compose 생성 — 기존 폐기 결정 유지. 해석 패널은 **결정적 파싱**만.
- 붙여넣기(팀원 설정 가져오기) 경로의 구조 변경 — 문구·스킨만 정비(이미 가장 매끄러운 경로).
- OS 네이티브 파일 픽커(브라우저 제약) — 탐색기는 폴백으로 유지만.

## 구조

### R1. 진입

- 헤더에 상시 `[+ 등록]` 버튼(스위처 밖). 프로젝트 0개면 본문 빈 상태 대문: "프로젝트를 등록하고 시작하세요" + 두 CTA — **[팀원 설정 받았어요]**(붙여넣기) / **[처음 설정해요]**(워크벤치).
- 경로 선택: 폴더 탐색 대신 ① **git 레포 후보 목록**(관례 루트 `~/IdeaProjects`·`~/projects`·최근 사용 경로를 1단계 스캔, `.git`/compose 뱃지 표시 — 신규 API `GET /api/repo-candidates`) ② 경로 직접 붙여넣기 입력 ③ 기존 탐색기(폴백, 접힘). 후보 스캔도 **버튼으로 실행**(자동 아님), 스캔 범위를 화면에 명시.

### R2. 워크벤치 (위저드 대체) — 좌 폼 · 우 compose 전문 · 양방향 연동 (형 확정)

한 화면 2열, 스텝 없음. **우측 = compose 전문(SoT)**, **좌측 = 그 문서를 쉽게 만지는 렌즈**.

- **우: compose 전문 에디터** — 기존 하이라이트/행번호 유지 + ① 빈 시작 시 주석 풍부한 표준 템플릿 프리로드 ② 인라인 검증(디바운스로 compose_validate 실행, 에러↔행 연결 거터 표시) ③ 필수값 `???` 마커(검증이 잔존 `???` 차단).
- **좌 상단: 서비스 재료** — 스캔 결과를 근거와 함께 나열, 클릭 = 우측에 블록 삽입(compose-scaffold 재사용) + 출처 주석.
  - `🐳 <서브레포>` "+ 서비스로" — 근거: `<경로>/Dockerfile · EXPOSE n · ARG X*(필수)`
  - `🧩 <서브레포>` "+ include로" — 근거: 자체 compose 보유 (항상 보이게 — 조용한 누락 금지)
  - `📄 기존 compose` "불러오기" — 시작점 로드(덮어쓰기 confirm 유지)
- **좌 하단: marina 옵션 폼** — x-marina 각 키가 행동어 라벨+한줄 설명(R4 사전)이 붙은 **기입 공간**으로 상시 노출:
  - `node_modules 재사용` — 패턴 태그 입력(기본 제안 node_modules·.venv)
  - `내 컴퓨터의 DB/Redis 쓰기` — 포트→host 행 추가(6379/3306/5432 원클릭 + 직접 입력)
  - `브라우저 주소 자동 발급` — 서비스 체크(우측 services 파싱으로 목록 자동 갱신)
  - `이미지 빌드 전에 미리 빌드` / `JDK 버전` / `빌드 변수(필수 * 표시)` — 동형
- **양방향 연동 규칙 (결정적)**:
  - 폼 ↔ YAML 은 **x-marina 서브트리만** 바인딩 — 폼 수정 → `serialize_xmarina` 로 x-marina 블록만 서지컬 교체(`_edit_xmarina_block` — services 쪽 사용자 주석 보존, 이미 존재), YAML 수정 → `parse_xmarina` 로 폼 갱신. 왕복 불변 파서가 이미 있어 드리프트 없음.
  - services 는 YAML 이 유일 SoT — 폼은 읽기만(게이트웨이 서비스 목록 등). 
  - 폼이 모르는 x-marina 키는 **보존** + "그 외 설정 n개(YAML 참조)" 표기 — 폼이 문서를 훼손하지 않는다.
  - YAML 의 x-marina 가 파싱 불가면 폼을 잠그고 "YAML 을 고치면 다시 열려요" 안내(조용한 덮어쓰기 금지).
- **하단 바**: 검증 상태(✓ 문법 OK / ⚠ n건) + `[검증 후 등록]`.
- 상태 유실 제거: 스텝 소멸 + **Esc/닫기 = 초안 자동 보관**(localStorage, 재진입 시 "이어서 작성" 제안).

### R3. 서비스 해석 줄 — compose 를 몰라도 읽게 (형: "도커 컴포즈 자체를 이해 못하는 사람들")

x-marina 는 좌측 폼이 곧 해석이므로(R2), 해석은 **services 쪽**만 담당한다:

- 에디터 위 토글 `[YAML | 해석]`: services 를 결정적 파싱으로 사람 문장 번역한 읽기 전용 뷰.
  - `web: ./web 폴더의 Dockerfile 로 빌드해서 포트 3000 으로 서빙`
  - `be-api: BUILD_ENV 값이 필요해요 (아직 비어 있음)`
  - `include: ai-api 의 자체 compose 를 통째로 가져옴`
- 지원 키만 번역(build/ports/expose/environment/env_file/depends_on/include), 모르는 키는 "그 외 설정 n개(YAML 참조)"로 정직하게. 신규 API 불필요 — 프론트 파싱(표시 전용).

### R4. x-marina 옵션 사전 — 알아먹는 언어 (형: "게이트웨이·포워드 알아먹게")

모든 노출 지점(재료 스니펫·해석 패널·치트시트)이 같은 사전을 쓴다:

| 키 | 라벨(행동어) | 한 줄 효과 | 비유/예시 |
|---|---|---|---|
| `links` | node_modules 재사용 | 워크트리를 새로 만들어도 의존성 재설치를 안 함 | "무거운 짐은 본가에 두고 열쇠만 복사" |
| `forward` | 내 컴퓨터의 DB/Redis 쓰기 | 컨테이너 안 `localhost:6379` 가 내 맥의 Redis 로 연결됨 | "컨테이너에 구멍을 뚫어 호스트로" |
| `gateway` | 브라우저 주소 자동 발급 | 워크트리마다 `이름.프로젝트.localhost` 주소가 생김 — 포트 몰라도 됨 | "지점마다 간판 달아주기" |
| `prebuild` | 이미지 빌드 전에 미리 빌드 | jar 처럼 먼저 구워야 하는 산출물을 자동으로 | |
| `java` | JDK 버전 지정 | 서브레포별 JDK 를 sdkman 에서 골라 씀 | |
| `build.args` | 빌드 변수 | Dockerfile 의 ARG 값 채우기 — `*` 는 필수 | |

- `[표준 안내]` 버튼 = 이 사전 + compose 최소 문법(services/build/ports 3개) 치트시트를 워크벤치 안 패널로(문서 링크 아님).
- UI 문구 규칙: 원어(`x-marina.forward`)는 모노 보조 표기로만, 라벨은 항상 행동어.

### R5. 완료·연결

- 등록 성공 = 모달 닫힘 → 새 프로젝트 자동 선택(현행) + **해당 카드 하이라이트(2s) + 카드에 `▶ 전체 시작` 상시 노출**(첫 시작 전 한정 — D7 토글 규칙의 빈 상태와 동일하므로 모순 없음) + 빌드 로그 탭 안내 토스트("시작하면 빌드 진행이 여기 보여요").
- alert/confirm 전면 제거 → 인라인 배너·토스트(Orca 톤). 자동 시작 없음(붙여넣기 경로의 옵트인만 유지).

### R6. Orca 스킨 재도색 범위

- `#registerBackdrop` 전체(entry/paste/워크벤치), links 모달, 서비스 구성(ⓘ) 모달, act-menu 와 동일 토큰: 무테두리 패널·`--st-*`·`--sys-*` 정리(레거시 `--muted`/`--accent` 인라인 제거), 모노 코드 블록, hover 액션.
- 위저드 코드(app-2 renderWiz*)는 워크벤치 완성 후 **삭제**(죽은 코드 금지). `/api/compose-scan`·`scaffold` 는 재료 서랍이 그대로 사용.

## 기존 API 매핑 (신규 최소)

| 기능 | API | 상태 |
|---|---|---|
| git 레포 후보 목록 | `GET /api/repo-candidates` | **신규** (관례 루트+최근 경로 1단계 스캔) |
| 재료 스캔 | `/api/compose-scan` | 재사용 |
| 블록 삽입 | `/api/compose-scaffold` | 재사용 |
| 기존 compose 찾기 | `/api/compose-detect` | 재사용 (깊이 1 한계 명시 문구) |
| 인라인 검증 | `/api/compose-validate` | **신규 노출**(기존 compose_validate 함수를 등록 없이 단독 호출) |
| 등록 | `/api/compose-register` | 재사용 |
| 붙여넣기 | `/api/compose-import` | 재사용 |

## 안정화 (구 위저드 급소 — 구조로 흡수)

- Esc 유실 → 초안 자동 보관. 뒤로가기 유실 → 스텝 소멸. 고급 편도행 → 워크벤치가 곧 에디터.
- 등록 버튼 위치 → 헤더 상시. `composeViewConfig` 신규 중 실패 → 워크벤치에선 해석 패널이 대체.
- add-project(infer 체크리스트) 잔재 → subrepos 편집(⚙) 전용으로 분리 유지, 등록 모달에서 UI 조각 제거.

## 테스트·검증

- 단위: repo-candidates(스캔 범위·뱃지), compose-validate 단독 엔드포인트, 해석 패널 번역기(지원 키 전수 + 미지원 키 정직 폴백 + `???` 마커), 초안 보관/복원.
- UI 스모크: 워크벤치 구조 불변식(서랍·에디터·검증바 존재, 위저드 잔재 부재), 기존 등록 e2e(test-compose-register-api) 회귀.
- 실브라우저(Aside): 신규 등록 해피패스(레포 후보 → 재료 2개 삽입 → `???` 채움 → 검증 → 등록 → 카드 하이라이트 → ▶ 시작), 붙여넣기 경로 회귀, 해석 패널 번역 정확성, 다크/라이트.

## 구현 순서 (플랜에서 상세화)

1. Orca 스킨 토큰 정리 + 모달 재도색(구조 변경 없이 먼저 — 즉시 체감)
2. 워크벤치 골격(진입·2열·에디터 이전) + 초안 보관
3. 재료 서랍(스캔 재사용 + 근거 표기 + 스니펫 사전 R4)
4. 해석 패널(R3) + 인라인 검증(compose-validate 노출)
5. repo-candidates + 진입 개편(R1) + 완료 연결(R5)
6. 위저드 제거 + 회귀·실브라우저 검증
